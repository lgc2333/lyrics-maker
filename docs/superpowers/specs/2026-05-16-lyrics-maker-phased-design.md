# 歌词打轴软件分阶段设计（基础设施优先）

## 1. 问题与目标

项目目标是构建一款 Web 端歌词打轴软件，最终覆盖时轴编辑、歌词打轴、导入导出与快捷键自定义等完整能力。  
当前采用“基础设施优先”路线：先建立可持续扩展的底层，再分阶段挂接高价值功能，避免后续返工。

已确认的阶段 1 约束：

- 只做编辑基础设施，不做时轴实际功能
- 多语言首发仅 `zh-CN`，但框架必须可扩展
- 撤销/重做覆盖阶段 1 全部功能，并为未来动作预留扩展
- 快捷键只做注册框架，不做重绑 UI
- 持久化采用 File System Access API，优先 Chrome/Edge；不支持浏览器可提示不支持
- 交付可运行的最小 UI 骨架 + 核心状态管理

## 2. 分阶段交付规划

### Phase 1：基础设施基座

- 统一工程数据模型（项目、歌词行、词块、配置）
- Command Dispatcher + Undo/Redo 历史栈（可扩展）
- i18n 框架（首发 `zh-CN`）
- 快捷键注册框架（冲突检测、动作映射）
- 文件持久化（打开文件句柄、脏状态、`Ctrl+S` 写盘）
- 最小 UI 骨架与状态联动

### Phase 2：音频与时轴核心

- 音频导入、播放/暂停、当前时间同步
- TAP BPM、Offset、拍号设置
- Timing Points（新增/编辑/删除）
- 节拍器与播放时间精确同步（含下一拍点 `latch` 收尾提示音规则）
- 音乐音量与音效音量独立控制

### Phase 3：主视图与网格系统

- 波形/频谱切换显示
- 主视图高度缩放、自动居中/滚动
- 网格线、播放头、分隔线同步滚动
- 吸附精度体系与网格映射
- 时间线右侧控制区：缩放按钮 + 分隔线密度切换（common：1/4、1/8…；triplets：1/3、1/6…）

### Phase 4：歌词打轴能力

- 歌词编辑、自动切词、手动切词、批量切词
- WordTimelineBar 三态词块与拖拽边界
- 逐词/逐句打轴流程（`D` / `Shift+D` / `Enter` 等）

### Phase 5：导入导出与完善

- TXT/LRC/ASS/JSON 导入导出插件化
- 快捷键自定义 UI
- 多语言扩展、可用性优化、测试补齐

## 3. 架构设计

采用三层架构：

1. **Core（纯业务层）**
   - `domain`：数据模型与规则（与 Vue 无关）
   - `commands`：所有编辑动作统一命令化（`execute` / `undo` / `redo`）
2. **Platform / Infrastructure（平台与适配）**
   - `i18n` 注册与文案装配
   - 快捷键注册中心与冲突检测
   - 文件读写与保存编排（File System Access API）
3. **App/UI（应用与展示）**
   - Pinia Store 承载会话状态
   - 最小布局壳（Menu、Transport、MainView、ModePanel）

关键边界约束：

- UI 不直接改数据，必须通过 Command Dispatcher
- 时间内部统一使用秒（浮点），吸附只在交互层生效
- 持久化与业务模型解耦，支持未来替换后端

## 4. 数据流与错误处理

### 数据流

1. 用户操作（按钮/快捷键）触发 Action Intent
2. Dispatcher 组装并执行 Command
3. Command 更新 domain state，写入历史栈
4. Store 响应状态变化，驱动 UI
5. 保存时检查脏状态并序列化写盘

### 错误处理

- 文件读写失败：明确反馈错误，不静默吞掉
- 浏览器不支持 File System Access：明确提示能力受限
- Command 执行失败：拒绝入栈并返回错误原因
- 快捷键冲突：拒绝后注册项并记录冲突信息

## 5. 测试与验收

### 第一阶段测试范围

- 单元测试（核心）
  - Command 执行/撤销/重做链路
  - 历史栈边界（清空重做栈、批处理、失败回滚）
  - 持久化服务（序列化、脏状态、写盘失败）
- 关键交互测试（少量）
  - 最小 UI 骨架渲染
  - 快捷键注册与触发
  - `Ctrl+S` 保存流程（支持环境）

### 第一阶段验收标准

- 阶段 1 所有可编辑动作都支持撤销/重做
- 可创建并修改基础工程状态，且可写回文件
- 最小 UI 骨架可运行并反映状态变化
- 测试通过并能作为 Phase 2 基座

## 6. 本设计的范围控制

本设计刻意不在 Phase 1 落地时轴/音频实际功能，只提供后续能力所需的稳定底层。  
通过先固化数据模型、命令体系和持久化机制，后续时轴与歌词功能可按增量方式接入，降低跨阶段改造成本。
